Deployment (การเผยแพร่เว็บ)

🟡 Intermediate ⏱️ อ่าน ~10 นาที
📋 Prerequisite
✓ Git + GitHub
✓ Vercel
✓ Supabase

สรุป Workflow ตั้งแต่เขียนโค้ดจนเว็บขึ้นออนไลน์ พร้อมข้อควรระวังที่สำคัญ

📋 ขั้นตอน Deploy ครบวงจร

1️⃣ เขียนโค้ด → ทดสอบใน localhost
⬇️
2️⃣ git add . → git commit → git push
⬇️
3️⃣ Vercel ดึงโค้ดจาก GitHub → Build อัตโนมัติ
⬇️
4️⃣ ตรวจ Build Logs — ผ่านหรือไม่?
⬇️
5️⃣ ✅ เว็บออนไลน์! หรือ ❌ แก้ Error แล้ว Push ใหม่

🚨 ข้อควรระวังสำคัญ 15 ข้อ

อ่านทุกข้อก่อนเปิดใช้งานเว็บจริง!

1. ❌ ห้าม Push .env

ความเสี่ยง: API Key, รหัสผ่าน ถูกเปิดเผยต่อสาธารณะ
ป้องกัน: ใส่ .env ใน .gitignore ก่อน Commit แรกเสมอ
แก้ไข: เปลี่ยน Key ทันที → git rm --cached .env

2. ❌ ห้ามใช้ service_role key ใน Frontend

ความเสี่ยง: ข้าม RLS ได้ทั้งหมด → ใครก็เข้าถึงข้อมูลทุกอย่างได้
ป้องกัน: ห้ามมี prefix NEXT_PUBLIC_ กับ service_role key
แก้ไข: ลบออกจาก Client code ทันที + เปลี่ยน Key

3. ❌ ห้ามปิด RLS โดยไม่จำเป็น

ความเสี่ยง: ผู้ใช้ทุกคนเห็น/แก้/ลบข้อมูลของทุกคนได้
ป้องกัน: เปิด RLS ทุกตาราง + ตั้ง Policy ครบ 4 แบบ

4. ❌ ห้ามใช้ git reset --hard โดยไม่เข้าใจ

ความเสี่ยง: ลบโค้ดทั้งหมดอย่างถาวร กู้คืนไม่ได้
ป้องกัน: ใช้ --soft แทน หรือ git revert ที่ปลอดภัยกว่า

5. ❌ ห้าม Force Push บน Branch หลัก

ความเสี่ยง: เขียนทับประวัติของทีม → โค้ดคนอื่นหาย
ป้องกัน: ห้ามใช้ git push --force บน main

6. ❌ ห้ามรัน DELETE/DROP โดยไม่ตรวจ WHERE

ความเสี่ยง: ลบข้อมูลทั้งตารางหรือลบตารางทั้งตัว
ป้องกัน: ตรวจ WHERE ทุกครั้ง ทดสอบใน Development ก่อน มี Backup

7. ❌ ห้าม Copy โค้ดจาก AI โดยไม่อ่าน

ความเสี่ยง: โค้ดอาจมี bug, ช่องโหว่ security, หรือ pattern ที่ล้าสมัย
ป้องกัน: อ่านทุกบรรทัด, Commit ก่อนแก้ใหญ่, ทดสอบใน Local

8. ⚠️ ห้ามส่ง Secret ให้ AI

ห้ามวาง API Key, Database Password, service_role key ให้ AI ช่วยดู ใช้ค่าตัวอย่างแทน

9. ⚠️ ห้ามเก็บ Password แบบ Plain Text

Supabase Auth จัดการ hashing ให้แล้ว อย่าเก็บรหัสผ่านเองในตารางแยก

10. ⚠️ ห้ามแก้ Production โดยตรง

แก้ใน Local → ทดสอบ → Push → ให้ Vercel Deploy อัตโนมัติ อย่าแก้โค้ดผ่าน Dashboard

11. ⚠️ ห้ามเปลี่ยน Schema บน Production โดยไม่มีแผน

ทดสอบ SQL ใน Development ก่อน ถ้าจะเพิ่ม/ลบ Column ต้องมั่นใจว่าโค้ดรองรับแล้ว

12. ⚠️ ห้ามใช้ Package ที่ไม่รู้แหล่งที่มา

ตรวจ npm package ก่อนติดตั้ง: ดูจำนวนดาวน์โหลด, ผู้สร้าง, อายุ, GitHub issues

13. ⚠️ ห้ามเปิด Permission กว้างเกินไป

RLS Policy ควรจำกัดเฉพาะสิ่งที่จำเป็น — ใช้หลัก Least Privilege

14. ⚠️ ตรวจ Redirect URL ทุกครั้ง

ต้องเพิ่มทั้ง localhost + production URL ใน Supabase Auth → URL Config ไม่งั้น Login ไม่กลับมาที่เว็บ

15. ⚠️ Redeploy หลังเปลี่ยน Env Vars

เปลี่ยน Environment Variables บน Vercel แล้วต้อง Deploy ใหม่เสมอ — ไม่งั้นยังใช้ค่าเก่าอยู่!

✅ Checklist ก่อน Deploy

⚠️ สิ่งที่คนมักลืม

1) Env Vars ใน Vercel ไม่ตรงกับ Local 2) ลืม Redeploy หลังเปลี่ยน Env Vars 3) Redirect URL ไม่ได้เพิ่ม production domain 4) RLS Policy ยังไม่ครบ 4 แบบ
