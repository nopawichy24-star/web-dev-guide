คู่มือ Git ฉบับละเอียด

🟢 Beginner ⏱️ อ่าน ~20 นาที
📋 Prerequisite
✓ ติดตั้ง Git แล้ว
✓ พื้นฐาน Command Line

Git คือระบบที่ช่วยบันทึกทุกการเปลี่ยนแปลงของโค้ด เหมือน "ประวัติการแก้ไข" ใน Google Docs แต่ทรงพลังกว่ามาก

💡 ทำไมต้องใช้ Git?

ลองนึกว่าเราเขียนรายงานแล้วแก้ไปแก้มา จนไม่รู้ว่าเวอร์ชันไหนดีที่สุด Git ช่วยให้เรา ย้อนกลับ ไปเวอร์ชันไหนก็ได้ ดูว่าใครแก้อะไร เมื่อไหร่ และ ทำงานพร้อมกัน หลายคนโดยไม่ชนกัน

🆚 Git กับ GitHub ต่างกันอย่างไร?

📦 Git

โปรแกรมที่ติดตั้งในเครื่องคอมฯ ทำงานแบบ offline ได้

🔧 เป็นเครื่องมือ

🐙 GitHub

เว็บไซต์ที่เก็บโค้ดบนคลาวด์ ต้องonline

☁️ เป็นบริการ (เหมือน Google Drive)

🧩 แนวคิดหลักของ Git

📁 Repository (Repo)

คือโฟลเดอร์โปรเจกต์ที่ Git ติดตามอยู่ ข้างในมีโฟลเดอร์ซ่อน .git ที่เก็บประวัติทั้งหมด

⚠️ อย่าลบโฟลเดอร์ .git!

ถ้าลบ .git ประวัติทุกอย่างจะหายหมด! โฟลเดอร์นี้ซ่อนอยู่ (ขึ้นต้นด้วยจุด) ปกติจะมองไม่เห็น

🔄 3 สถานะของไฟล์ใน Git

ไฟล์ใน Git มี 3 สถานะ ลองนึกภาพว่าเหมือนการจัดของใส่กล่องส่งไปรษณีย์:

📂 Working Directory
โฟลเดอร์ทำงาน — แก้ไขไฟล์ตามใจ 🖊️
⬇️ git add
📋 Staging Area
พื้นที่เตรียม — เลือกไฟล์ที่จะบันทึก 📝
⬇️ git commit
📸 Repository
ที่เก็บถาวร — บันทึกเรียบร้อย ปลอดภัย! ✅
💜 เปรียบเทียบให้เห็นภาพ

🖊️ Working Directory = โต๊ะทำงาน (แก้ไฟล์ไปเรื่อยๆ)
📋 Staging Area = จัดของใส่กล่อง (เลือกว่าจะส่งอะไร)
📸 Commit = ส่งไปรษณีย์แล้ว (บันทึกถาวร ย้อนดูได้)

📸 Commit

คือ "จุดบันทึก" หนึ่งจุด เก็บ snapshot ของไฟล์ทั้งหมด ณ ตอนนั้น แต่ละ Commit มี:

  • 🔑 Hash ID — รหัสเฉพาะ (เช่น a1b2c3d)
  • 📝 Commit Message — ข้อความอธิบายว่าเปลี่ยนอะไร
  • 👤 ชื่อคนที่ทำ + วันเวลา

🌿 Branch

คือ "เส้นทางแยก" ของโค้ด ให้เราทดลองเพิ่มฟีเจอร์ใหม่โดยไม่กระทบโค้ดหลัก

diagram
main:     ●───●───●───●───●  (โค้ดหลัก เสถียร)
               \       /
feature:        ●───●──┘  (ทดลองฟีเจอร์ใหม่ แล้ว merge กลับ)

🔀 Merge

คือการนำโค้ดจาก Branch หนึ่งไปรวมกับอีก Branch เช่น รวม feature กลับเข้า main

💥 Conflict

เกิดเมื่อ 2 คน (หรือ 2 Branch) แก้ไขบรรทัดเดียวกันในไฟล์เดียวกัน Git ไม่รู้จะเอาของใคร เราต้องเลือกเอง

☁️ Remote Repository

คือ Repository ที่อยู่บนเว็บ (เช่น GitHub) เราส่งโค้ดจากเครื่องเราขึ้นไปด้วย git push และดึงลงมาด้วย git pull

⚡ คำสั่ง Git ทั้งหมดที่ต้องรู้

1. git --version — ตรวจสอบเวอร์ชัน

ใช้ทำอะไร: ดูว่าติดตั้ง Git แล้วหรือยัง และเวอร์ชันอะไร

bash💻 Terminal
git --version

ผลลัพธ์: git version 2.47.1

⚠️ ถ้าขึ้น "command not found"

แปลว่ายังไม่ได้ติดตั้ง Git กลับไปดูบท 🔧 ติดตั้งเครื่องมือ

2. git config — ตั้งค่า Git

ใช้ทำอะไร: บอก Git ว่าเราชื่ออะไร ใช้อีเมลอะไร (ทำครั้งเดียว)

bash💻 Terminal
# ตั้งชื่อ (เปลี่ยนเป็นชื่อจริงของคุณ)
git config --global user.name "Nopawich"

# ตั้งอีเมล (ใช้อีเมลเดียวกับ GitHub)
git config --global user.email "your@email.com"

# ดูค่าทั้งหมดที่ตั้งไว้
git config --list

3. git init — เริ่มโปรเจกต์ Git ใหม่

ใช้ทำอะไร: เปลี่ยนโฟลเดอร์ธรรมดาให้กลายเป็น Git Repository

bash💻 Terminal — ต้อง cd เข้าไปในโฟลเดอร์โปรเจกต์ก่อน
# สร้างโฟลเดอร์ใหม่และเข้าไป
mkdir my-project
cd my-project

# เริ่ม Git ในโฟลเดอร์นี้
git init

ผลลัพธ์: Initialized empty Git repository in /path/my-project/.git/

🚨 ข้อผิดพลาดที่พบบ่อย

อย่ารัน git init ในโฟลเดอร์ Home หรือ Desktop! ต้องรันในโฟลเดอร์โปรเจกต์เฉพาะเจาะจง ไม่งั้น Git จะติดตามไฟล์ทุกอย่างในเครื่อง

4. git status — ดูสถานะไฟล์ ⭐

ใช้ทำอะไร: ดูว่าไฟล์ไหนเปลี่ยนแปลง ไฟล์ไหนพร้อมบันทึก — คำสั่งที่ใช้บ่อยที่สุด!

bash💻 Terminal
git status

ผลลัพธ์ตัวอย่าง:

output
On branch main

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   index.html    ← ไฟล์ที่แก้ไขแล้ว (สีแดง)

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        style.css                 ← ไฟล์ใหม่ที่ยังไม่ได้ track
💜 นิสัยที่ดี

รัน git status ก่อนและหลังทุกคำสั่ง Git เพื่อดูว่าเกิดอะไรขึ้น!

5. git add — เลือกไฟล์ที่จะบันทึก ⭐

ใช้ทำอะไร: ย้ายไฟล์จาก Working Directory ไปที่ Staging Area

bash💻 Terminal
# เพิ่มไฟล์เฉพาะตัว
git add index.html

# เพิ่มหลายไฟล์
git add index.html style.css

# เพิ่มทุกไฟล์ที่เปลี่ยนแปลง (ใช้บ่อยสุด)
git add .

# เพิ่มเฉพาะไฟล์นามสกุล .html
git add *.html
⚠️ ระวังก่อน git add .

คำสั่ง git add . จะเพิ่มทุกไฟล์ รวมถึงไฟล์ที่อาจมี Secret! ใช้ git status ตรวจสอบก่อนเสมอ และตั้ง .gitignore ให้เรียบร้อย

6. git commit — บันทึกการเปลี่ยนแปลง ⭐

ใช้ทำอะไร: สร้าง "จุดบันทึก" ถาวร จากไฟล์ที่อยู่ใน Staging Area

bash💻 Terminal
# บันทึกพร้อมข้อความอธิบาย
git commit -m "add login page"

# ลัดรวม add + commit (เฉพาะไฟล์ที่ track แล้ว ไม่รวมไฟล์ใหม่)
git commit -am "fix typo in header"

ผลลัพธ์:

output
[main a1b2c3d] add login page
 2 files changed, 45 insertions(+), 3 deletions(-)

7. git log — ดูประวัติ Commit

ใช้ทำอะไร: ดูรายการ Commit ทั้งหมดที่เคยทำ

bash💻 Terminal
# ดูประวัติเต็ม
git log

# ดูแบบย่อ 1 บรรทัดต่อ Commit (แนะนำ!)
git log --oneline

# ดู 5 อันล่าสุด
git log --oneline -5

# กด q เพื่อออกจาก log

ผลลัพธ์ git log --oneline:

output
a1b2c3d (HEAD -> main) add login page
e4f5g6h fix typo in header
i7j8k9l initial commit

8. git diff — ดูว่าเปลี่ยนอะไรบ้าง

ใช้ทำอะไร: เปรียบเทียบไฟล์ปัจจุบันกับ Commit ก่อนหน้า

bash💻 Terminal
# ดูสิ่งที่เปลี่ยนแต่ยังไม่ได้ add
git diff

# ดูสิ่งที่ add แล้วแต่ยังไม่ commit
git diff --staged

# ดูเฉพาะไฟล์
git diff index.html

9. git branch — จัดการ Branch

ใช้ทำอะไร: สร้าง, ดูรายชื่อ, ลบ Branch

bash💻 Terminal
# ดู Branch ทั้งหมด (* คือ Branch ที่อยู่ตอนนี้)
git branch

# สร้าง Branch ใหม่
git branch feature-login

# ลบ Branch (ต้อง merge แล้ว)
git branch -d feature-login

# ลบ Branch แบบบังคับ (ระวัง! โค้ดอาจหาย)
git branch -D feature-login

# เปลี่ยนชื่อ Branch ปัจจุบัน
git branch -M main

10. git switch — สลับ Branch (แนะนำ)

ใช้ทำอะไร: ย้ายไปทำงานใน Branch อื่น

bash💻 Terminal
# สลับไป Branch ที่มีอยู่แล้ว
git switch feature-login

# สร้าง Branch ใหม่ + สลับไปเลย
git switch -c feature-signup

# กลับไป main
git switch main

11. git checkout — สลับ Branch (แบบเก่า)

ใช้ทำอะไร: เหมือน git switch แต่เป็นคำสั่งเก่า ยังใช้ได้

bash💻 Terminal
# สลับ Branch
git checkout feature-login

# สร้าง + สลับ
git checkout -b feature-signup
💜 แนะนำ

ใช้ git switch แทน git checkout เพราะชัดเจนกว่า — checkout ทำได้หลายอย่างเกินไปจนสับสนได้

12. git merge — รวม Branch

ใช้ทำอะไร: นำโค้ดจาก Branch หนึ่งมารวมกับ Branch ปัจจุบัน

bash💻 Terminal
# 1. สลับไป Branch ที่จะรับ (เช่น main)
git switch main

# 2. รวม Branch feature เข้ามา
git merge feature-login

ผลลัพธ์ที่ดี: Merge made by the 'ort' strategy.

13. git remote — เชื่อมกับ GitHub

ใช้ทำอะไร: ตั้งค่าที่อยู่ Remote Repository

bash💻 Terminal
# เพิ่ม Remote ชื่อ origin (ทำครั้งเดียวต่อโปรเจกต์)
git remote add origin https://github.com/username/my-project.git

# ดู Remote ทั้งหมด
git remote -v

# เปลี่ยน URL
git remote set-url origin https://github.com/username/new-url.git

# ลบ Remote
git remote remove origin

14. git clone — ดาวน์โหลดโปรเจกต์จาก GitHub

ใช้ทำอะไร: คัดลอกโปรเจกต์ทั้งหมดจาก GitHub มาที่เครื่องเรา

bash💻 Terminal
git clone https://github.com/username/project-name.git

จะได้โฟลเดอร์ project-name พร้อมประวัติ Git ทั้งหมด

15. git pull — ดึงโค้ดล่าสุดจาก GitHub ⭐

ใช้ทำอะไร: ดึงการเปลี่ยนแปลงล่าสุดจาก GitHub มาอัปเดตในเครื่องเรา

bash💻 Terminal
git pull
💜 ควรทำก่อน push เสมอ!

รัน git pull ก่อน git push ทุกครั้ง เพื่อหลีกเลี่ยง Conflict

16. git fetch — ตรวจสอบอัปเดตโดยไม่รวม

ใช้ทำอะไร: ดาวน์โหลดข้อมูลใหม่จาก GitHub แต่ยังไม่รวมเข้ากับโค้ดเรา (ปลอดภัยกว่า pull)

bash💻 Terminal
# ดึงข้อมูลมาดู
git fetch

# จะรวมเมื่อไหร่ ค่อยสั่ง merge เอง
git merge origin/main

17. git push — ส่งโค้ดขึ้น GitHub ⭐

ใช้ทำอะไร: อัปโหลด Commit จากเครื่องเราไปที่ GitHub

bash💻 Terminal
# Push ครั้งแรก (ตั้ง upstream)
git push -u origin main

# Push ครั้งต่อไป (สั้นลง)
git push
🚨 ก่อน Push ทุกครั้ง!

ตรวจสอบว่าไม่มีไฟล์ .env, API Key, รหัสผ่าน หรือข้อมูลลับใดๆ อยู่ในไฟล์ที่จะ Push! ใช้ git status ตรวจสอบก่อน

18. git reset — ย้อนกลับ (ระวัง!)

ใช้ทำอะไร: ยกเลิก Commit หรือเอาไฟล์ออกจาก Staging Area

bash💻 Terminal
# เอาไฟล์ออกจาก Staging (ปลอดภัย ไม่เสียข้อมูล)
git reset index.html

# ย้อนกลับ 1 Commit แต่เก็บไฟล์ไว้ (ปลอดภัย)
git reset --soft HEAD~1

# ย้อนกลับ 1 Commit เอาออกจาก staging ด้วย (ปลอดภัย)
git reset HEAD~1

# ⚠️ ย้อนกลับ + ลบการเปลี่ยนแปลงทั้งหมด (อันตราย!)
git reset --hard HEAD~1
🚨 อันตราย: git reset --hard

--hard จะลบโค้ดที่เปลี่ยนแปลงอย่างถาวร! กู้คืนไม่ได้ ห้ามใช้ถ้ายังไม่เข้าใจ ใช้ --soft แทนจะปลอดภัยกว่า

19. git restore — กู้ไฟล์คืน

ใช้ทำอะไร: เอาไฟล์กลับไปสภาพ Commit ล่าสุด (ยกเลิกการแก้ไข)

bash💻 Terminal
# ยกเลิกการแก้ไขไฟล์ (กลับไปเหมือน Commit ล่าสุด)
git restore index.html

# เอาไฟล์ออกจาก Staging (เหมือน git reset แต่ชัดกว่า)
git restore --staged index.html

20. git revert — ย้อน Commit แบบปลอดภัย

ใช้ทำอะไร: สร้าง Commit ใหม่ที่ "ทำตรงข้าม" กับ Commit ที่ระบุ — ปลอดภัยกว่า reset เพราะไม่ลบประวัติ

bash💻 Terminal
# ย้อน Commit ล่าสุด
git revert HEAD

# ย้อน Commit ที่ระบุ Hash
git revert a1b2c3d
💜 reset vs revert

reset = ลบ Commit ทิ้ง (เหมือนฉีกหน้าสมุดออก) — อันตราย
revert = สร้าง Commit ใหม่ที่แก้กลับ (เหมือนเขียนแก้ไขหน้าใหม่) — ปลอดภัย ✅

21. git stash — เก็บงานชั่วคราว

ใช้ทำอะไร: เก็บงานที่ยังไม่เสร็จไว้ชั่วคราว เพื่อไปทำอย่างอื่นก่อน แล้วค่อยกลับมาทำต่อ

bash💻 Terminal
# เก็บงานปัจจุบันไว้ชั่วคราว
git stash

# ดูรายการ stash ทั้งหมด
git stash list

# เอางานกลับมาทำต่อ
git stash pop

# ลบ stash ทั้งหมด
git stash clear

✍️ วิธีเขียน Commit Message ที่ดี

✅ ตัวอย่างที่ดี

bash
git commit -m "add: user registration form"
git commit -m "fix: login button not responding on mobile"
git commit -m "update: change header color to blue"
git commit -m "remove: unused CSS file"
git commit -m "refactor: simplify database query"

❌ ตัวอย่างที่ไม่ดี

bash
git commit -m "update"           ← อัปเดตอะไร?
git commit -m "fix bug"          ← บั๊กอะไร?
git commit -m "asdf"             ← ไม่มีความหมาย
git commit -m "."                ← ?????

🌿 Branch Workflow

โปรเจกต์ที่เป็นระเบียบจะแบ่ง Branch ดังนี้:

Branchหน้าที่ตัวอย่างชื่อ
mainโค้ดหลักที่พร้อมใช้งาน — ห้ามพัง!main
developโค้ดพัฒนา รวมฟีเจอร์ก่อนเข้า maindevelop
featureเพิ่มฟีเจอร์ใหม่feature/login
bugfixแก้บั๊กbugfix/fix-404
hotfixแก้ไขเร่งด่วนบน productionhotfix/fix-crash

ตัวอย่างการใช้ Feature Branch

bash💻 Terminal
# 1. สร้าง Branch ใหม่จาก main
git switch main
git switch -c feature/login-page

# 2. ทำงาน... แก้ไฟล์... commit ไปเรื่อยๆ
git add .
git commit -m "add: login form UI"
git commit -m "add: form validation"

# 3. เสร็จแล้ว กลับไป main
git switch main

# 4. รวม feature เข้า main
git merge feature/login-page

# 5. ลบ Branch ที่ใช้เสร็จแล้ว
git branch -d feature/login-page

# 6. Push ขึ้น GitHub
git push

💥 แก้ Merge Conflict ทีละขั้นตอน

สถานการณ์ตัวอย่าง

สมมติ Branch main แก้ไฟล์ index.html บรรทัดที่ 5 เป็น "สวัสดี" และ Branch feature แก้บรรทัดเดียวกันเป็น "Hello" — เมื่อ Merge จะเกิด Conflict

ขั้นตอนแก้ไข

1️⃣ เมื่อสั่ง Merge แล้วเกิด Conflict จะเห็น:

output
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

2️⃣ เปิดไฟล์ที่มี Conflict จะเห็นเครื่องหมายพิเศษ:

html📄 index.html
<<<<<<< HEAD
<h1>สวัสดี</h1>
=======
<h1>Hello</h1>
>>>>>>> feature
💡 อ่านเครื่องหมาย

<<<<<<< HEAD = โค้ดของ Branch ปัจจุบัน (main)
======= = เส้นแบ่ง
>>>>>>> feature = โค้ดของ Branch ที่จะ Merge เข้ามา

3️⃣ เลือกว่าจะเอาแบบไหน แล้วลบเครื่องหมายทิ้ง:

html📄 index.html — หลังแก้ Conflict
<h1>สวัสดี — Hello</h1>

4️⃣ บันทึกการแก้ไข:

bash💻 Terminal
git add index.html
git commit -m "resolve: merge conflict in index.html"
✅ จบ! Conflict ถูกแก้แล้ว

Conflict ดูน่ากลัวแต่จริงๆ แค่เลือกโค้ดที่ถูกต้องแล้วลบเครื่องหมายทิ้ง ใน VS Code มีปุ่มช่วย "Accept Current / Accept Incoming / Accept Both" ให้กดง่ายๆ
