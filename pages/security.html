Security & Row Level Security (RLS)

🟢 Beginner ⏱️ อ่าน ~12 นาที
📋 Prerequisite
✓ Supabase Database
✓ Authentication

RLS คือระบบที่สำคัญที่สุดใน Supabase — มันกำหนดว่าผู้ใช้คนไหนเข้าถึงข้อมูลแถวไหนได้บ้าง

🚨 ถ้าไม่เปิด RLS จะเกิดอะไรขึ้น?

anon key อยู่ในโค้ด Frontend ที่ทุกคนเห็น → ถ้าไม่มี RLS → ใครก็อ่าน/แก้/ลบข้อมูลทั้งหมดได้! ผู้ใช้ A เห็นข้อมูลของผู้ใช้ B ได้ หรือลบข้อมูลทั้งตารางได้เลย!

🔑 auth.uid() คืออะไร?

auth.uid() คือฟังก์ชันของ Supabase ที่คืนค่า User ID ของคนที่กำลัง Login อยู่ — เป็นหัวใจของ RLS!

💡 แนวคิดง่ายๆ

auth.uid() = user_id หมายความว่า "ผู้ใช้ที่ login อยู่ ต้องเป็นเจ้าของข้อมูลแถวนั้น" — ถ้า user_id ในแถวไม่ตรงกับคนที่ login → ปฏิเสธ!

📝 สร้าง Policy ทีละตัว

Policy 1: อ่านข้อมูลตัวเอง

sql⚡ Supabase → SQL Editor
CREATE POLICY "ผู้ใช้อ่าน Task ของตัวเองได้"
  ON tasks FOR SELECT
  USING (auth.uid() = user_id);

Policy 2: เพิ่มข้อมูลตัวเอง

sql⚡ SQL Editor
CREATE POLICY "ผู้ใช้สร้าง Task ของตัวเองได้"
  ON tasks FOR INSERT
  WITH CHECK (auth.uid() = user_id);

Policy 3: แก้ไขข้อมูลตัวเอง

sql⚡ SQL Editor
CREATE POLICY "ผู้ใช้แก้ไข Task ของตัวเองได้"
  ON tasks FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

Policy 4: ลบข้อมูลตัวเอง

sql⚡ SQL Editor
CREATE POLICY "ผู้ใช้ลบ Task ของตัวเองได้"
  ON tasks FOR DELETE
  USING (auth.uid() = user_id);

🚀 รวมทั้ง 4 Policy ในคำสั่งเดียว

sql⚡ SQL Editor — Copy ไปรันทีเดียวจบ!
-- เปิด RLS
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

-- SELECT: อ่านเฉพาะของตัวเอง
CREATE POLICY "tasks_select" ON tasks FOR SELECT
  USING (auth.uid() = user_id);

-- INSERT: สร้างเฉพาะของตัวเอง
CREATE POLICY "tasks_insert" ON tasks FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- UPDATE: แก้เฉพาะของตัวเอง
CREATE POLICY "tasks_update" ON tasks FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

-- DELETE: ลบเฉพาะของตัวเอง
CREATE POLICY "tasks_delete" ON tasks FOR DELETE
  USING (auth.uid() = user_id);

🔐 anon key vs service_role key (ทบทวน)

🟡 anon key

ใช้ใน Frontend ได้ ✅

ต้องมี RLS ป้องกัน!

ทำได้เฉพาะที่ Policy อนุญาต

🔴 service_role key

ใช้ฝั่ง Server เท่านั้น ❌

ข้าม RLS ได้ทั้งหมด!

เข้าถึง/แก้/ลบข้อมูลทุกอย่างได้

⚠️ คำเตือนสำคัญ

🚨 ห้ามทำสิ่งเหล่านี้!
  • ❌ ห้ามใส่ service_role key ในโค้ด Frontend
  • ❌ ห้าม Push service_role key ขึ้น GitHub
  • ❌ ห้ามปิด RLS โดยไม่เข้าใจผลกระทบ
  • ❌ ห้ามเปิดเผย Database Password
  • ❌ ห้ามสร้าง Policy USING (true) สำหรับ DELETE/UPDATE (= ใครก็ลบ/แก้ได้!)

💥 ตัวอย่างความเสียหายถ้า Policy ผิด

Policy ที่ผิดความเสียหาย
ไม่เปิด RLSทุกคนอ่าน/แก้/ลบข้อมูลทุกอย่างได้
SELECT ใช้ USING (true)ทุกคนเห็นข้อมูลของทุกคน (รวมข้อมูลส่วนตัว)
DELETE ใช้ USING (true)ใครก็ลบข้อมูลของคนอื่นได้!
service_role key อยู่ใน Frontendข้าม RLS ทั้งหมด = เหมือนไม่มี RLS
🧠 ทดสอบความเข้าใจ RLS
auth.uid() = user_id หมายความว่าอะไร?
👆 กดเพื่อดูคำตอบ
🔑 "คนที่ Login อยู่ ต้องเป็นเจ้าของข้อมูลแถวนั้น" — ถ้า user_id ไม่ตรง → ปฏิเสธการเข้าถึง
USING vs WITH CHECK ต่างกันอย่างไร?
👆 กดเพื่อดูคำตอบ
USING = ตรวจข้อมูลที่มีอยู่แล้ว (อ่าน/แก้/ลบ) / WITH CHECK = ตรวจข้อมูลใหม่ที่จะเพิ่ม/แก้
ทำไม USING (true) ใน DELETE ถึงอันตราย?
👆 กดเพื่อดูคำตอบ
🚨 เพราะ true = อนุญาตทุกคน → ใครก็ลบข้อมูลของใครก็ได้! ต้องใช้ auth.uid() = user_id
