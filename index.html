🏗️ ส่วนประกอบของเว็บไซต์

1. Frontend (ส่วนหน้าบ้าน)

คือทุกสิ่งที่ผู้ใช้เห็นและโต้ตอบได้บนหน้าจอ ประกอบด้วย 3 ภาษาหลัก:

ภาษาหน้าที่เปรียบเทียบ
HTML โครงสร้างเนื้อหา — กำหนดว่ามีอะไรบ้าง 🦴 โครงกระดูกของร่างกาย
CSS รูปลักษณ์ — กำหนดหน้าตา สี ขนาด 👔 เสื้อผ้าและแต่งหน้า
JavaScript พฤติกรรม — ทำให้เว็บตอบสนอง 🧠 สมองและกล้ามเนื้อ

ตัวอย่าง HTML ง่ายๆ:

HTML 📄 ไฟล์ index.html 
<!DOCTYPE html>
<html lang="th">
<head>
  <title>เว็บไซต์แรกของฉัน</title>
</head>
<body>
  <h1>สวัสดี!</h1>
  <p>นี่คือเว็บไซต์แรกของฉัน</p>
  <button onclick="alert('คลิกแล้ว!')">กดฉัน</button>
</body>
</html>

2. Backend (ส่วนหลังบ้าน)

คือส่วนที่ทำงานเบื้องหลัง ผู้ใช้ไม่เห็นโดยตรง ทำหน้าที่:

  • รับข้อมูลจากผู้ใช้ (เช่น ฟอร์มสมัครสมาชิก)
  • ตรวจสอบสิทธิ์ (เช่น รหัสผ่านถูกไหม?)
  • อ่านและเขียนข้อมูลในฐานข้อมูล
  • ส่งอีเมล, ประมวลผลรูปภาพ ฯลฯ
💡 สำหรับคู่มือนี้

เราจะใช้ Supabase เป็น Backend ไม่ต้องเขียน Backend เอง — ช่วยให้ผู้เริ่มต้นสร้างเว็บแอปได้เร็วขึ้นมาก!

3. Database (ฐานข้อมูล)

คือที่เก็บข้อมูลทั้งหมดของเว็บไซต์ เช่น:

  • ข้อมูลสมาชิก (ชื่อ, อีเมล, รหัสผ่าน)
  • บทความ, โพสต์, ความคิดเห็น
  • รายการสินค้า, คำสั่งซื้อ

4. Hosting (ที่พักเว็บไซต์)

เว็บไซต์ต้อง "อาศัยอยู่" ที่ไหนสักที่บนอินเทอร์เน็ต เพื่อให้คนอื่นเข้าถึงได้

ในคู่มือนี้เราใช้ Vercel เป็น Hosting ซึ่งจะ auto-deploy ทุกครั้งที่เราส่งโค้ดไปที่ GitHub

⚙️ Backend

🗄️ Database

เก็บข้อมูลผู้ใช้ใหม่ → พร้อมให้เรียกใช้ทีหลัง

🏨 Hosting

ทำให้ทั้งหมดเข้าถึงได้ผ่านอินเทอร์เน็ต 24 ชม.

🆚 Static Website vs Dynamic Website

📄 Static Website

เนื้อหาเหมือนกันทุกคน ไม่มีระบบสมาชิก ไม่มีฐานข้อมูล

ตัวอย่าง: หน้าเว็บประวัติส่วนตัว, บล็อกส่วนตัว

⚡ Dynamic Website

เนื้อหาเปลี่ยนตามผู้ใช้ มีระบบ Login มีฐานข้อมูล

ตัวอย่าง: Facebook, Twitter, ร้านค้าออนไลน์

✅ สิ่งที่ควรเข้าใจแล้วตอนนี้

เว็บไซต์ = Frontend (หน้าตา) + Backend (เบื้องหลัง) + Database (ข้อมูล) + Hosting (ที่พัก)

🔗 API คืออะไร?

API คือ "พนักงานเสิร์ฟ" ระหว่าง Frontend กับ Database:

🖥️ Frontend (ลูกค้า)
"ขอรายการ Task ทั้งหมด"
⬇️ ส่ง Request
🔗 API (พนักงานเสิร์ฟ)
รับคำสั่ง → ตรวจสิทธิ์ → ดึงข้อมูล
⬇️ ⬆️
🗄️ Database (ห้องครัว)
เก็บข้อมูลจริง → ส่งกลับ
⬆️ ส่ง Response
🖥️ Frontend แสดงผลให้ผู้ใช้เห็น

📡 HTTP Request — วิธีที่ Browser คุยกับ Server

เมื่อเว็บต้องการข้อมูลจาก Server จะส่ง HTTP Request มี 4 แบบหลัก:

Methodความหมายใช้เมื่อSupabase
🟢 GETขอข้อมูล (อ่าน)โหลดรายการ Task.select()
🔵 POSTส่งข้อมูลใหม่ (สร้าง)สร้าง Task ใหม่.insert()
🟡 PUT / PATCHแก้ไขข้อมูลเปลี่ยนชื่อ Task.update()
🔴 DELETEลบข้อมูลลบ Task.delete()
💡 ข่าวดี!

function Counter() {

  const [count, setCount] = useState(0)

      <button onClick={() => setCount(count + 1)}>

      </button>
    </div>
  )
  // เมื่อ setCount ถูกเรียก → count เปลี่ยน → React re-render → หน้าจออัปเดต!
}

9️⃣ useEffect — ทำงานหลัง Render ⭐

ใช้ทำงานที่ต้องเกิดหลังจากหน้าจอแสดงผลแล้ว เช่น โหลดข้อมูลจาก Supabase

tsx📄 ตัวอย่าง useEffect — โหลดข้อมูลตอนเปิดหน้า
import { useState, useEffect } from 'react'
import { supabase } from '@/lib/supabase'

function TaskPage() {
  const [tasks, setTasks] = useState([])
  const [loading, setLoading] = useState(true)

  // useEffect รันหลังจาก Component แสดงผลครั้งแรก
  useEffect(() => {
    async function loadTasks() {
      const { data } = await supabase.from('tasks').select('*')
      if (data) setTasks(data)
      setLoading(false)
    }
    loadTasks()
  }, [])  // ← [] = รันครั้งเดียวตอนเปิดหน้า

  if (loading) return <p>⏳ กำลังโหลด...</p>

  return (
    <ul>
      {tasks.map(t => <li key={t.id}>{t.title}</li>)}
    </ul>
  )
}
Dependency Arrayรันเมื่อตัวอย่าง
[]ครั้งเดียว ตอนเปิดหน้าโหลดข้อมูลครั้งแรก
[count]ทุกครั้งที่ count เปลี่ยนอัปเดตเมื่อค่าเปลี่ยน
ไม่ใส่ทุกครั้งที่ re-render ⚠️ระวัง! อาจทำงานซ้ำไม่หยุด
❌ Common Mistake — Infinite Loop!

ถ้าลืมใส่ [] ใน useEffect + มี setState ข้างใน → จะวน loop ไม่หยุด! เพราะ setState → re-render → useEffect รันใหม่ → setState → re-render → ...

🔟 Context — แชร์ข้อมูลทั้งแอป

ปกติต้องส่ง Props จากแม่ → ลูก → หลาน ทีละชั้น (เรียก Prop Drilling) แต่ Context ให้ทุก Component เข้าถึงข้อมูลได้เลย!

tsx📄 ตัวอย่าง Context — แชร์ข้อมูล User ทั้งแอป
import { createContext, useContext, useState } from 'react'

// 1. สร้าง Context
const UserContext = createContext(null)

// 2. สร้าง Provider (ครอบทั้งแอป)
function UserProvider({ children }) {
  const [user, setUser] = useState(null)
  return (
    <UserContext.Provider value={{ user, setUser }}>
      {children}
    </UserContext.Provider>
  )
}

// 3. ใช้ใน Component ไหนก็ได้!
function Navbar() {
  const { user } = useContext(UserContext)
  return <p>สวัสดี {user?.name || 'Guest'}</p>
}

function LoginButton() {
  const { setUser } = useContext(UserContext)
  return (
    <button onClick={() => setUser({ name: 'สมชาย' })}>
      Login
    </button>
  )
}

ใช้ทำอะไร: ดูว่าไฟล์ไหนเปลี่ยนแปลง ไฟล์ไหนพร้อมบันทึก — คำสั่งที่ใช้บ่อยที่สุด!

bash💻 Terminal
git status

ผลลัพธ์ตัวอย่าง:

output
On branch main

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   index.html    ← ไฟล์ที่แก้ไขแล้ว (สีแดง)

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        style.css                 ← ไฟล์ใหม่ที่ยังไม่ได้ track
💜 นิสัยที่ดี

รัน git status ก่อนและหลังทุกคำสั่ง Git เพื่อดูว่าเกิดอะไรขึ้น!

5. git add — เลือกไฟล์ที่จะบันทึก ⭐

ใช้ทำอะไร: ย้ายไฟล์จาก Working Directory ไปที่ Staging Area

bash💻 Terminal
# เพิ่มไฟล์เฉพาะตัว
git add index.html

# เพิ่มหลายไฟล์
git add index.html style.css

# เพิ่มทุกไฟล์ที่เปลี่ยนแปลง (ใช้บ่อยสุด)
git add .

# เพิ่มเฉพาะไฟล์นามสกุล .html
git add *.html
⚠️ ระวังก่อน git add .

คำสั่ง git add . จะเพิ่มทุกไฟล์ รวมถึงไฟล์ที่อาจมี Secret! ใช้ git status ตรวจสอบก่อนเสมอ และตั้ง .gitignore ให้เรียบร้อย

6. git commit — บันทึกการเปลี่ยนแปลง ⭐

ใช้ทำอะไร: สร้าง "จุดบันทึก" ถาวร จากไฟล์ที่อยู่ใน Staging Area

bash💻 Terminal
# บันทึกพร้อมข้อความอธิบาย
git commit -m "add login page"

# ลัดรวม add + commit (เฉพาะไฟล์ที่ track แล้ว ไม่รวมไฟล์ใหม่)
git commit -am "fix typo in header"

ผลลัพธ์:

output
[main a1b2c3d] add login page
 2 files changed, 45 insertions(+), 3 deletions(-)

git diff

9. git branch — จัดการ Branch

ใช้ทำอะไร: สร้าง, ดูรายชื่อ, ลบ Branch

bash💻 Terminal
# ดู Branch ทั้งหมด (* คือ Branch ที่อยู่ตอนนี้)
git branch

# สร้าง Branch ใหม่
git branch feature-login

# ลบ Branch (ต้อง merge แล้ว)
git branch -d feature-login

# ลบ Branch แบบบังคับ (ระวัง! โค้ดอาจหาย)
git branch -D feature-login

git branch -M main

10. git switch — สลับ Branch (แนะนำ)

ใช้ทำอะไร: ย้ายไปทำงานใน Branch อื่น

bash💻 Terminal
# สลับไป Branch ที่มีอยู่แล้ว
git switch feature-login

# สร้าง Branch ใหม่ + สลับไปเลย
git switch -c feature-signup

# กลับไป main
git switch main

11. git checkout — สลับ Branch (แบบเก่า)

ใช้ทำอะไร: เหมือน git switch แต่เป็นคำสั่งเก่า ยังใช้ได้

bash💻 Terminal
# สลับ Branch
git checkout feature-login

# สร้าง + สลับ
git checkout -b feature-signup
💜 แนะนำ

ใช้ git switch แทน git checkout เพราะชัดเจนกว่า — checkout ทำได้หลายอย่างเกินไปจนสับสนได้

12. git merge — รวม Branch

ใช้ทำอะไร: นำโค้ดจาก Branch หนึ่งมารวมกับ Branch ปัจจุบัน

bash💻 Terminal
# 1. สลับไป Branch ที่จะรับ (เช่น main)
git switch main

# 2. รวม Branch feature เข้ามา
git merge feature-login

ผลลัพธ์ที่ดี: Merge made by the 'ort' strategy.

14. git clone — ดาวน์โหลดโปรเจกต์จาก GitHub

ใช้ทำอะไร: คัดลอกโปรเจกต์ทั้งหมดจาก GitHub มาที่เครื่องเรา

bash💻 Terminal
git clone https://github.com/username/project-name.git

จะได้โฟลเดอร์ project-name พร้อมประวัติ Git ทั้งหมด

15. git pull — ดึงโค้ดล่าสุดจาก GitHub ⭐

ใช้ทำอะไร: ดึงการเปลี่ยนแปลงล่าสุดจาก GitHub มาอัปเดตในเครื่องเรา

bash💻 Terminal
git pull
💜 ควรทำก่อน push เสมอ!

รัน git pull ก่อน git push ทุกครั้ง เพื่อหลีกเลี่ยง Conflict

16. git fetch — ตรวจสอบอัปเดตโดยไม่รวม

ใช้ทำอะไร: ดาวน์โหลดข้อมูลใหม่จาก GitHub แต่ยังไม่รวมเข้ากับโค้ดเรา (ปลอดภัยกว่า pull)

bash💻 Terminal
# ดึงข้อมูลมาดู
git fetch

# จะรวมเมื่อไหร่ ค่อยสั่ง merge เอง
git merge origin/main

17. git push — ส่งโค้ดขึ้น GitHub ⭐

ใช้ทำอะไร: อัปโหลด Commit จากเครื่องเราไปที่ GitHub

bash💻 Terminal
# Push ครั้งแรก (ตั้ง upstream)
git push -u origin main

# Push ครั้งต่อไป (สั้นลง)
git push
🚨 ก่อน Push ทุกครั้ง!

ตรวจสอบว่าไม่มีไฟล์ .env, API Key, รหัสผ่าน หรือข้อมูลลับใดๆ อยู่ในไฟล์ที่จะ Push! ใช้ git status ตรวจสอบก่อน

18. git reset — ย้อนกลับ (ระวัง!)

ใช้ทำอะไร: ยกเลิก Commit หรือเอาไฟล์ออกจาก Staging Area

bash💻 Terminal
# เอาไฟล์ออกจาก Staging (ปลอดภัย ไม่เสียข้อมูล)
git reset index.html

# ย้อนกลับ 1 Commit แต่เก็บไฟล์ไว้ (ปลอดภัย)
git reset --soft HEAD~1

# ย้อนกลับ 1 Commit เอาออกจาก staging ด้วย (ปลอดภัย)
git reset HEAD~1

# ⚠️ ย้อนกลับ + ลบการเปลี่ยนแปลงทั้งหมด (อันตราย!)
git reset --hard HEAD~1
🚨 อันตราย: git reset --hard

--hard จะลบโค้ดที่เปลี่ยนแปลงอย่างถาวร! กู้คืนไม่ได้ ห้ามใช้ถ้ายังไม่เข้าใจ ใช้ --soft แทนจะปลอดภัยกว่า

19. git restore — กู้ไฟล์คืน

ใช้ทำอะไร: เอาไฟล์กลับไปสภาพ Commit ล่าสุด (ยกเลิกการแก้ไข)

bash💻 Terminal
# ยกเลิกการแก้ไขไฟล์ (กลับไปเหมือน Commit ล่าสุด)
git restore index.html

# เอาไฟล์ออกจาก Staging (เหมือน git reset แต่ชัดกว่า)
git restore --staged index.html

20. git revert — ย้อน Commit แบบปลอดภัย

ใช้ทำอะไร: สร้าง Commit ใหม่ที่ "ทำตรงข้าม" กับ Commit ที่ระบุ — ปลอดภัยกว่า reset เพราะไม่ลบประวัติ

bash💻 Terminal
# ย้อน Commit ล่าสุด
git revert HEAD

# ย้อน Commit ที่ระบุ Hash
git revert a1b2c3d
💜 reset vs revert

reset = ลบ Commit ทิ้ง (เหมือนฉีกหน้าสมุดออก) — อันตราย
revert = สร้าง Commit ใหม่ที่แก้กลับ (เหมือนเขียนแก้ไขหน้าใหม่) — ปลอดภัย ✅

21. git stash — เก็บงานชั่วคราว

ใช้ทำอะไร: เก็บงานที่ยังไม่เสร็จไว้ชั่วคราว เพื่อไปทำอย่างอื่นก่อน แล้วค่อยกลับมาทำต่อ

bash💻 Terminal
# เก็บงานปัจจุบันไว้ชั่วคราว
git stash

# ดูรายการ stash ทั้งหมด
git stash list

# เอางานกลับมาทำต่อ
git stash pop

# ลบ stash ทั้งหมด
git stash clear

🌿 Branch Workflow

โปรเจกต์ที่เป็นระเบียบจะแบ่ง Branch ดังนี้:

Branchหน้าที่ตัวอย่างชื่อ
mainโค้ดหลักที่พร้อมใช้งาน — ห้ามพัง!main
developโค้ดพัฒนา รวมฟีเจอร์ก่อนเข้า maindevelop
featureเพิ่มฟีเจอร์ใหม่feature/login
bugfixแก้บั๊กbugfix/fix-404
hotfixแก้ไขเร่งด่วนบน productionhotfix/fix-crash

ตัวอย่างการใช้ Feature Branch

bash💻 Terminal
# 1. สร้าง Branch ใหม่จาก main
git switch main
git switch -c feature/login-page

# 2. ทำงาน... แก้ไฟล์... commit ไปเรื่อยๆ
git add .
git commit -m "add: login form UI"
git commit -m "add: form validation"

# 3. เสร็จแล้ว กลับไป main
git switch main

# 4. รวม feature เข้า main
git merge feature/login-page

# 5. ลบ Branch ที่ใช้เสร็จแล้ว
git branch -d feature/login-page

# 6. Push ขึ้น GitHub
git push

💥 แก้ Merge Conflict ทีละขั้นตอน

สถานการณ์ตัวอย่าง

สมมติ Branch main แก้ไฟล์ index.html บรรทัดที่ 5 เป็น "สวัสดี" และ Branch feature แก้บรรทัดเดียวกันเป็น "Hello" — เมื่อ Merge จะเกิด Conflict

ขั้นตอนแก้ไข

1️⃣ เมื่อสั่ง Merge แล้วเกิด Conflict จะเห็น:

output
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

2️⃣ เปิดไฟล์ที่มี Conflict จะเห็นเครื่องหมายพิเศษ:

html📄 index.html
<<<<<<< HEAD
<h1>สวัสดี</h1>
=======
<h1>Hello</h1>
>>>>>>> feature
💡 อ่านเครื่องหมาย

<<<<<<< HEAD = โค้ดของ Branch ปัจจุบัน (main)
======= = เส้นแบ่ง
>>>>>>> feature = โค้ดของ Branch ที่จะ Merge เข้ามา

3️⃣ เลือกว่าจะเอาแบบไหน แล้วลบเครื่องหมายทิ้ง:

html📄 index.html — หลังแก้ Conflict
<h1>สวัสดี — Hello</h1>

4️⃣ บันทึกการแก้ไข:

bash💻 Terminal
git add index.html
git commit -m "resolve: merge conflict in index.html"
✅ จบ! Conflict ถูกแก้แล้ว

Conflict ดูน่ากลัวแต่จริงๆ แค่เลือกโค้ดที่ถูกต้องแล้วลบเครื่องหมายทิ้ง ใน VS Code มีปุ่มช่วย "Accept Current / Accept Incoming / Accept Both" ให้กดง่ายๆ

🔗 เชื่อม Git ในเครื่องกับ GitHub

มี 2 วิธีหลัก:

วิธีที่ 1: HTTPS (แนะนำสำหรับมือใหม่ ✅)

bash💻 Terminal — ในโฟลเดอร์โปรเจกต์
# 1. เชื่อม Git กับ GitHub Repository
git remote add origin https://github.com/username/my-project.git

# 2. ตั้งชื่อ Branch หลักเป็น main
git branch -M main

# 3. Push โค้ดขึ้น GitHub ครั้งแรก
git push -u origin main
💡 เปลี่ยน username เป็นชื่อ GitHub ของคุณ!

URL จะอยู่ในหน้า Repository บน GitHub → กดปุ่ม Code สีเขียว → คัดลอก HTTPS URL มาวาง

เมื่อ Push ครั้งแรก GitHub จะถาม Authentication — ต้องใช้ Personal Access Token แทนรหัสผ่าน:

📌 สร้าง Personal Access Token

  1. เข้า GitHub → กดรูปโปรไฟล์มุมบนขวา → Settings
  2. เลื่อนลงซ้ายสุด → Developer settings
  3. เลือก Personal access tokens → Tokens (classic)
  4. กด Generate new token (classic)
  5. ตั้งชื่อ (เช่น "my-laptop") ตั้ง Expiration
  6. ติ๊ก repo (อย่างน้อย)
  7. กด Generate token
  8. คัดลอก Token ทันที! จะเห็นแค่ครั้งเดียว
🚨 สำคัญมาก!

Token นี้คือกุญแจเข้าบัญชี GitHub ของคุณ ห้ามแชร์ ห้ามวางในโค้ด ห้าม Push ขึ้น GitHub! เก็บไว้ที่ปลอดภัย (เช่น Notes ที่ล็อกรหัส)

วิธีที่ 2: SSH (สะดวกกว่าเมื่อตั้งค่าแล้ว)

bash💻 Terminal
# 1. สร้าง SSH Key (กด Enter ไปเรื่อยๆ ไม่ต้องตั้งรหัส)
ssh-keygen -t ed25519 -C "your@email.com"

# 2. คัดลอก Public Key
# Mac:
cat ~/.ssh/id_ed25519.pub | pbcopy
# Windows (PowerShell):
Get-Content ~/.ssh/id_ed25519.pub | clip

# 3. ไปวางที่ GitHub → Settings → SSH and GPG keys → New SSH key

# 4. ทดสอบการเชื่อมต่อ
ssh -T git@github.com

ผลลัพธ์ที่ดี: Hi username! You've successfully authenticated

สมมติว่ามีโปรเจกต์ในเครื่องแล้ว และสร้าง Repo เปล่าบน GitHub แล้ว:

🔧 ปัญหาที่พบบ่อยกับ GitHub

ปัญหาสาเหตุวิธีแก้
Permission denied ยังไม่ได้ตั้ง Authentication หรือ Token หมดอายุ สร้าง Personal Access Token ใหม่ หรือตั้ง SSH Key
Authentication failed ใช้รหัสผ่าน GitHub โดยตรง (GitHub ไม่รับแล้ว) ต้องใช้ Personal Access Token แทน
Repository not found URL ผิด หรือ Repo เป็น Private แต่ไม่มีสิทธิ์ ตรวจสอบ URL ให้ตรง ดูด้วย git remote -v
remote origin already exists เคย add remote แล้ว ใช้ git remote set-url origin URL_ใหม่
rejected (non-fast-forward) GitHub มีโค้ดใหม่ที่เราไม่มี รัน git pull --rebase ก่อนแล้ว Push ใหม่
large file error ไฟล์ใหญ่เกิน 100MB ใส่ไฟล์ใหญ่ใน .gitignore หรือใช้ Git LFS

git push

📊 สร้าง Table (ตาราง)

ลองนึกว่า Table คือตาราง Excel แต่ละแผ่น:

คำศัพท์เทียบกับ Excelตัวอย่าง
Tableแผ่นงาน (Sheet)ตาราง users, ตาราง tasks
Rowแถว (แต่ละรายการ)ผู้ใช้คนที่ 1, Task ที่ 1
Columnคอลัมน์ (หัวข้อ)ชื่อ, อีเมล, วันที่สร้าง
Primary Keyลำดับที่ (ไม่ซ้ำกัน)id
Foreign Keyลิงก์ไปอีก Sheetuser_id → users.id

สร้าง Table ผ่าน Dashboard

  1. เข้า Supabase Dashboard → Table Editor
  2. กด Create a new table
  3. ตั้งชื่อ (เช่น tasks)
  4. ✅ เปิด Enable Row Level Security (สำคัญมาก!)
  5. เพิ่ม Column ตามต้องการ
  6. กด Save

หรือสร้างด้วย SQL

ไปที่ SQL Editor แล้วรัน:

sql⚡ Supabase → SQL Editor
-- ตาราง profiles (ข้อมูลผู้ใช้)
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  username TEXT UNIQUE,
  full_name TEXT,
  avatar_url TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- ตาราง tasks (รายการงาน)
CREATE TABLE tasks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  title TEXT NOT NULL,
  completed BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- เปิด RLS ทั้ง 2 ตาราง (สำคัญมาก!)
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;
⚠️ เปิด RLS ทุกตารางเสมอ!

ถ้าไม่เปิด RLS ใครก็สามารถอ่าน/แก้/ลบข้อมูลทุกอย่างได้ผ่าน anon key ที่อยู่ในโค้ด Frontend! ดูรายละเอียดเรื่อง RLS ในบท 🛡️ Security

📝 SQL พื้นฐานที่ต้องรู้

อ่านข้อมูล

SELECT * FROM tasks;

-- อ่านเฉพาะบางคอลัมน์

SELECT * FROM tasks WHERE completed = false;

-- เรียงลำดับ
SELECT * FROM tasks ORDER BY created_at DESC;

-- จำกัดจำนวน
SELECT * FROM tasks LIMIT 10;

-- รวมเงื่อนไข
SELECT * FROM tasks
  WHERE user_id = 'xxx' AND completed = false
  ORDER BY created_at DESC
  LIMIT 20;

เพิ่มข้อมูล

แก้ไขข้อมูล

sql⚡ SQL Editor
UPDATE tasks
SET completed = true

ลบข้อมูล

sql⚡ SQL Editor
-- ลบแถวที่ระบุ

🚨 อันตราย! DELETE และ DROP

DELETE FROM tasks; (ไม่มี WHERE) จะลบข้อมูลทั้งตาราง!
DROP TABLE tasks; จะลบตารางทั้งตัว!
ตรวจสอบ WHERE ให้แน่ใจทุกครั้งก่อนรัน DELETE! และห้ามรัน DROP บน Production โดยไม่มี Backup!

⚡ Realtime — อัปเดตข้อมูลทันที

Realtime ทำให้เว็บอัปเดตข้อมูลทันทีที่มีการเปลี่ยนแปลง ไม่ต้อง refresh!

ts📄 ตัวอย่างโค้ด — ฟังการเปลี่ยนแปลง tasks
// ฟังเมื่อมีการเพิ่ม/แก้/ลบ Task
const channel = supabase
  .channel('tasks-changes')
  .on(
    'postgres_changes',
    {
      event: '*',          // ฟังทุก event (INSERT/UPDATE/DELETE)
      schema: 'public',
      table: 'tasks'
    },
    (payload) => {
      console.log('ข้อมูลเปลี่ยน:', payload)
      // โหลด Tasks ใหม่
      loadTasks()
    }
  )
  .subscribe()

// ยกเลิกการฟังเมื่อไม่ใช้แล้ว
// supabase.removeChannel(channel)

ts📄 ตัวอย่างโค้ดใน Component
async function createTask(title: string) {
  // ดึง user ปัจจุบัน
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) throw new Error('ยังไม่ได้เข้าสู่ระบบ')

  const { data, error } = await supabase
    .from('tasks')
    .insert({ title, user_id: user.id })
    .select()        // ให้ส่งข้อมูลที่สร้างกลับมา
    .single()        // รับแค่ 1 แถว

  if (error) {
    console.error('สร้าง Task ไม่สำเร็จ:', error.message)
    return null
  }
  return data
}

📖 Read — อ่านข้อมูล

ts📄 ตัวอย่างโค้ด
async function getTasks() {
  const { data, error } = await supabase
    .from('tasks')
    .select('*')
    .order('created_at', { ascending: false })

  if (error) {
    console.error('โหลด Tasks ไม่สำเร็จ:', error.message)
    return []
  }
  return data     // RLS จะกรองให้เห็นเฉพาะ Task ของ user นี้อัตโนมัติ!
}

async function toggleTask(id: string, completed: boolean) {
  const { error } = await supabase
    .from('tasks')
    .update({ completed: !completed })
    .eq('id', id)              // WHERE id = ...

  if (error) {
    console.error('แก้ไข Task ไม่สำเร็จ:', error.message)
  }
}

async function deleteTask(id: string) {
  // แนะนำ: ถามยืนยันก่อนลบเสมอ!

👂 ฟัง Session เปลี่ยนแปลง (onAuthStateChange)

ts📄 ใช้ใน layout หรือ component หลัก
import { useEffect } from 'react'
import { supabase } from '@/lib/supabase'
import { useRouter } from 'next/navigation'

function useAuth() {
  const router = useRouter()

  useEffect(() => {
    // ฟังทุกครั้งที่ Login/Logout
    const { data: { subscription } } = supabase.auth.onAuthStateChange(
      (event, session) => {
        if (event === 'SIGNED_IN') {
          router.push('/tasks')
        }
        if (event === 'SIGNED_OUT') {
          router.push('/login')
        }
      }
    )

    // cleanup เมื่อ component ถูกทำลาย
    return () => subscription.unsubscribe()
  }, [])
}

2️⃣ One-to-Many — ผู้ใช้หนึ่งคนมีหลาย Tasks ⭐

create table public.tasks (

  user_id uuid not null
    references public.profiles(id)
    on delete cascade,
  title text not null,
  is_done boolean default false,
  created_at timestamptz default now()
);

JOINคืนข้อมูลแบบไหนใช้เมื่อ

  tasks.id,

5️⃣ Index — ช่วยให้ค้นหาเร็วขึ้น

Index เปรียบเหมือนสารบัญของหนังสือ ฐานข้อมูลสามารถหาตำแหน่งข้อมูลได้เร็วขึ้นโดยไม่ต้องอ่านทุกแถว

สร้าง Index สำหรับคำสั่งที่ใช้บ่อย
-- ใช้บ่อยกับ WHERE user_id = ...
create index tasks_user_id_idx
on public.tasks(user_id);

-- ใช้บ่อยกับ ORDER BY created_at
create index tasks_created_at_idx
on public.tasks(created_at desc);

-- Index หลายคอลัมน์
create index tasks_user_created_idx
on public.tasks(user_id, created_at desc);

✅ ควรสร้าง

คอลัมน์ที่ใช้ใน WHERE, JOIN หรือ ORDER BY บ่อย และตารางมีข้อมูลจำนวนมาก

⚠️ ไม่ควรสร้างทุกคอลัมน์

Index ใช้พื้นที่เพิ่ม และทำให้ INSERT, UPDATE, DELETE ช้าลงเพราะต้องอัปเดต Index ด้วย

📤 อัปโหลดไฟล์

ts📄 ตัวอย่างโค้ด
async function uploadAvatar(file: File, userId: string) {
  // ตั้ง path เป็น userId/filename เพื่อจัดการ Policy ง่าย
  const filePath = `${userId}/${Date.now()}_${file.name}`

  const { error } = await supabase.storage
    .from('avatars')       // ชื่อ Bucket
    .upload(filePath, file, {
      cacheControl: '3600',  // cache 1 ชม.
      upsert: false          // false = ห้ามเขียนทับ
    })

  if (error) {
    console.error('อัปโหลดไม่สำเร็จ:', error.message)
    return null
  }

  // สร้าง URL สาธารณะ
  const { data } = supabase.storage
    .from('avatars')
    .getPublicUrl(filePath)

  return data.publicUrl
}

🛡️ Storage Policy

เหมือน RLS ใน Database — ต้องตั้ง Policy เพื่อกำหนดสิทธิ์:

sql⚡ SQL Editor
-- อนุญาตให้ผู้ใช้อัปโหลดเฉพาะในโฟลเดอร์ของตัวเอง
CREATE POLICY "users upload own folder"
  ON storage.objects FOR INSERT
  WITH CHECK (
    bucket_id = 'avatars'
    AND auth.uid()::text = (storage.foldername(name))[1]
  );

-- อนุญาตให้ทุกคนดูไฟล์ใน public bucket
CREATE POLICY "public read avatars"
  ON storage.objects FOR SELECT
  USING (bucket_id = 'avatars');

📤 ตัวอย่าง Upload Component

tsx📄 components/AvatarUpload.tsx
'use client'
import { useState } from 'react'
import { supabase } from '@/lib/supabase'

export default function AvatarUpload({ userId }: { userId: string }) {
  const [uploading, setUploading] = useState(false)

  async function handleUpload(e: React.ChangeEvent<HTMLInputElement>) {
    const file = e.target.files?.[0]
    if (!file) return

    // ตรวจขนาดไฟล์ (จำกัด 2MB)
    if (file.size > 2 * 1024 * 1024) {
      alert('ไฟล์ใหญ่เกิน 2MB!')
      return
    }

    // ตรวจประเภทไฟล์
    if (!file.type.startsWith('image/')) {
      alert('อัปโหลดได้เฉพาะรูปภาพ!')
      return
    }

    setUploading(true)
    const path = `${userId}/${Date.now()}.${file.name.split('.').pop()}`

    const { error } = await supabase.storage
      .from('avatars')
      .upload(path, file)

    if (error) alert('อัปโหลดไม่สำเร็จ: ' + error.message)
    else alert('อัปโหลดสำเร็จ! ✅')

    setUploading(false)
  }

  return (
    <label className="cursor-pointer">
      <span className="text-blue-600 hover:underline">
        {uploading ? '⏳ กำลังอัปโหลด...' : '📷 เปลี่ยนรูปโปรไฟล์'}
      </span>
      <input type="file" accept="image/*"
        onChange={handleUpload} disabled={uploading}
        className="hidden" />
    </label>
  )
}

Security & Row Level Security (RLS)

🟢 Beginner ⏱️ อ่าน ~24 นาที
📋 Prerequisite
✓ Supabase Database
✓ Authentication

RLS คือระบบที่สำคัญที่สุดใน Supabase — มันกำหนดว่าผู้ใช้คนไหนเข้าถึงข้อมูลแถวไหนได้บ้าง

🚨 ถ้าไม่เปิด RLS จะเกิดอะไรขึ้น?

anon key อยู่ในโค้ด Frontend ที่ทุกคนเห็น → ถ้าไม่มี RLS → ใครก็อ่าน/แก้/ลบข้อมูลทั้งหมดได้! ผู้ใช้ A เห็นข้อมูลของผู้ใช้ B ได้ หรือลบข้อมูลทั้งตารางได้เลย!

🧭 Authentication กับ Authorization ต่างกันอย่างไร?

สองคำนี้เกี่ยวกับความปลอดภัยเหมือนกัน แต่ทำหน้าที่คนละช่วงของระบบ

🔐 Authentication

ตรวจว่า “คุณคือใคร”

ตัวอย่าง: Login ด้วยอีเมลและรหัสผ่าน แล้ว Supabase คืนข้อมูลผู้ใช้และ Session

🛂 Authorization

ตรวจว่า “คุณทำอะไรได้บ้าง”

ตัวอย่าง: ผู้ใช้ที่ Login แล้วอ่านได้เฉพาะ Task ที่มี user_id ตรงกับตัวเอง

1. ผู้ใช้ Login
↓
2. Supabase ตรวจตัวตน
Authentication สำเร็จ → ได้ Session และ Access Token
↓
3. ผู้ใช้ขออ่านหรือแก้ข้อมูล
↓
4. RLS ตรวจสิทธิ์
Authorization ตรวจว่าแถวข้อมูลนั้นเป็นของผู้ใช้หรือไม่
💡 จำง่าย ๆ

Authentication = บัตรประชาชน ยืนยันว่าเป็นใคร
Authorization = บัตรผ่านประตู ระบุว่าเข้าโซนไหนได้

🎫 JWT ทำงานอย่างไร?

JWT มักถูกใช้เป็น Access Token หลัง Login โดยประกอบด้วย 3 ส่วนคั่นด้วยจุด

ส่วนเก็บอะไรข้อควรรู้
Headerชนิด Token และอัลกอริทึมลายเซ็นไม่ใช่ข้อมูลลับ
PayloadUser ID, Role, เวลาออกและเวลาหมดอายุอ่านได้ด้วยการ decode จึงห้ามเก็บรหัสผ่าน
Signatureลายเซ็นตรวจว่า Token ถูกแก้ไขหรือไม่ต้องตรวจด้วย Secret ฝั่งเซิร์ฟเวอร์
ตัวอย่างโครงสร้าง JWT Payload
{
  "sub": "user-uuid",
  "role": "authenticated",
  "iat": 1710000000,
  "exp": 1710003600
}
⚠️ JWT ไม่ใช่การเข้ารหัสข้อมูลลับ

ผู้ใช้สามารถ decode Header และ Payload ได้ สิ่งที่ทำให้ Token เชื่อถือได้คือ Signature ไม่ใช่การซ่อนข้อมูล

🔄 Session Management

Session คือสถานะการ Login ของผู้ใช้ ซึ่งโดยทั่วไปประกอบด้วย Access Token อายุสั้น และ Refresh Token ที่ใช้ขอ Access Token ใหม่

รายการหน้าที่แนวทางปลอดภัย
Access Tokenส่งไปกับคำขอ API เพื่อยืนยันผู้ใช้ควรมีอายุสั้นและตรวจวันหมดอายุ
Refresh Tokenใช้ต่ออายุ Session โดยไม่ต้อง Login ใหม่ต้องเก็บอย่างปลอดภัยและยกเลิกเมื่อ Logout
Sessionรวม Token และข้อมูลผู้ใช้ปัจจุบันฟังการเปลี่ยนแปลงเพื่ออัปเดต UI ให้ตรงสถานะจริง
ติดตามสถานะ Login ด้วย onAuthStateChange
const {
  data: { subscription }
} = supabase.auth.onAuthStateChange((event, session) => {
  if (event === 'SIGNED_IN') {
    console.log('Login แล้ว:', session.user.id)
  }

  if (event === 'SIGNED_OUT') {
    console.log('ออกจากระบบแล้ว')
  }

  updateNavigation(session?.user ?? null)
})

// เมื่อ Component ถูกทำลาย
subscription.unsubscribe()

✅ Best Practice — อย่าเชื่อ State ในหน้าเว็บอย่างเดียว

เมื่อเปิดเว็บใหม่หรือกลับมาจากแท็บอื่น ให้ตรวจ Session จริงจาก Supabase และใช้ onAuthStateChange เพื่อรองรับ Token ที่ถูกต่ออายุหรือผู้ใช้ Logout จากที่อื่น

🔑 auth.uid() คืออะไร?

auth.uid() คือฟังก์ชันของ Supabase ที่คืนค่า User ID ของคนที่กำลัง Login อยู่ — เป็นหัวใจของ RLS!

💡 แนวคิดง่ายๆ

auth.uid() = user_id หมายความว่า "ผู้ใช้ที่ login อยู่ ต้องเป็นเจ้าของข้อมูลแถวนั้น" — ถ้า user_id ในแถวไม่ตรงกับคนที่ login → ปฏิเสธ!

-- เปิด RLS
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

CREATE POLICY "tasks_select" ON tasks FOR SELECT
  USING (auth.uid() = user_id);

-- INSERT: สร้างเฉพาะของตัวเอง
CREATE POLICY "tasks_insert" ON tasks FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- UPDATE: แก้เฉพาะของตัวเอง
CREATE POLICY "tasks_update" ON tasks FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

-- DELETE: ลบเฉพาะของตัวเอง
CREATE POLICY "tasks_delete" ON tasks FOR DELETE
  USING (auth.uid() = user_id);

1. ❌ ห้าม Push .env

ความเสี่ยง: API Key, รหัสผ่าน ถูกเปิดเผยต่อสาธารณะ
ป้องกัน: ใส่ .env ใน .gitignore ก่อน Commit แรกเสมอ
แก้ไข: เปลี่ยน Key ทันที → git rm --cached .env

2. ❌ ห้ามใช้ service_role key ใน Frontend

ความเสี่ยง: ข้าม RLS ได้ทั้งหมด → ใครก็เข้าถึงข้อมูลทุกอย่างได้
ป้องกัน: ห้ามมี prefix NEXT_PUBLIC_ กับ service_role key
แก้ไข: ลบออกจาก Client code ทันที + เปลี่ยน Key

3. ❌ ห้ามปิด RLS โดยไม่จำเป็น

ความเสี่ยง: ผู้ใช้ทุกคนเห็น/แก้/ลบข้อมูลของทุกคนได้
ป้องกัน: เปิด RLS ทุกตาราง + ตั้ง Policy ครบ 4 แบบ

4. ❌ ห้ามใช้ git reset --hard โดยไม่เข้าใจ

ความเสี่ยง: ลบโค้ดทั้งหมดอย่างถาวร กู้คืนไม่ได้
ป้องกัน: ใช้ --soft แทน หรือ git revert ที่ปลอดภัยกว่า

5. ❌ ห้าม Force Push บน Branch หลัก

ความเสี่ยง: เขียนทับประวัติของทีม → โค้ดคนอื่นหาย
ป้องกัน: ห้ามใช้ git push --force บน main

6. ❌ ห้ามรัน DELETE/DROP โดยไม่ตรวจ WHERE

ความเสี่ยง: ลบข้อมูลทั้งตารางหรือลบตารางทั้งตัว
ป้องกัน: ตรวจ WHERE ทุกครั้ง ทดสอบใน Development ก่อน มี Backup

7. ❌ ห้าม Copy โค้ดจาก AI โดยไม่อ่าน

ความเสี่ยง: โค้ดอาจมี bug, ช่องโหว่ security, หรือ pattern ที่ล้าสมัย
ป้องกัน: อ่านทุกบรรทัด, Commit ก่อนแก้ใหญ่, ทดสอบใน Local

8. ⚠️ ห้ามส่ง Secret ให้ AI

ห้ามวาง API Key, Database Password, service_role key ให้ AI ช่วยดู ใช้ค่าตัวอย่างแทน

9. ⚠️ ห้ามเก็บ Password แบบ Plain Text

Supabase Auth จัดการ hashing ให้แล้ว อย่าเก็บรหัสผ่านเองในตารางแยก

10. ⚠️ ห้ามแก้ Production โดยตรง

แก้ใน Local → ทดสอบ → Push → ให้ Vercel Deploy อัตโนมัติ อย่าแก้โค้ดผ่าน Dashboard

11. ⚠️ ห้ามเปลี่ยน Schema บน Production โดยไม่มีแผน

ทดสอบ SQL ใน Development ก่อน ถ้าจะเพิ่ม/ลบ Column ต้องมั่นใจว่าโค้ดรองรับแล้ว

12. ⚠️ ห้ามใช้ Package ที่ไม่รู้แหล่งที่มา

ตรวจ npm package ก่อนติดตั้ง: ดูจำนวนดาวน์โหลด, ผู้สร้าง, อายุ, GitHub issues

13. ⚠️ ห้ามเปิด Permission กว้างเกินไป

RLS Policy ควรจำกัดเฉพาะสิ่งที่จำเป็น — ใช้หลัก Least Privilege

14. ⚠️ ตรวจ Redirect URL ทุกครั้ง

ต้องเพิ่มทั้ง localhost + production URL ใน Supabase Auth → URL Config ไม่งั้น Login ไม่กลับมาที่เว็บ

15. ⚠️ Redeploy หลังเปลี่ยน Env Vars

เปลี่ยน Environment Variables บน Vercel แล้วต้อง Deploy ใหม่เสมอ — ไม่งั้นยังใช้ค่าเก่าอยู่!

-- 2. ตาราง tasks
CREATE TABLE tasks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  title TEXT NOT NULL,
  completed BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 3. เปิด RLS
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

-- 4. Policy — profiles
CREATE POLICY "profiles_select" ON profiles FOR SELECT
  USING (auth.uid() = id);
CREATE POLICY "profiles_update" ON profiles FOR UPDATE
  USING (auth.uid() = id);

-- 5. Policy — tasks (ครบ 4 แบบ)
CREATE POLICY "tasks_select" ON tasks FOR SELECT
  USING (auth.uid() = user_id);
CREATE POLICY "tasks_insert" ON tasks FOR INSERT
  WITH CHECK (auth.uid() = user_id);
CREATE POLICY "tasks_update" ON tasks FOR UPDATE
  USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id);
CREATE POLICY "tasks_delete" ON tasks FOR DELETE
  USING (auth.uid() = user_id);

-- 6. สร้าง profile อัตโนมัติเมื่อ signup
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO profiles (id) VALUES (NEW.id);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION handle_new_user();

📄 middleware.ts — ป้องกันหน้าที่ต้อง Login ⭐

ts📄 middleware.ts — ไฟล์นี้อยู่ที่ root ของโปรเจกต์ (ไม่ใช่ใน app/)
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'
import { createClient } from '@supabase/supabase-js'

export async function middleware(request: NextRequest) {
  // หน้าที่ไม่ต้อง Login
  const publicPaths = ['/login']
  if (publicPaths.includes(request.nextUrl.pathname)) {
    return NextResponse.next()
  }

  // ตรวจ session จาก cookie
  const token = request.cookies.get('sb-access-token')?.value
  if (!token) {
    // ไม่มี session → ส่งไป login
    return NextResponse.redirect(new URL('/login', request.url))
  }

  return NextResponse.next()
}

export const config = {
  matcher: ['/tasks/:path*'],  // ใช้ middleware เฉพาะหน้า tasks
}

import { useState } from 'react'
import { supabase } from '@/lib/supabase'
import { useRouter } from 'next/navigation'

export default function LoginPage() {

  const [password, setPassword] = useState('')

      : await supabase.auth.signInWithPassword({ email, password })

    if (error) {
      setError(error.message)
    } else if (isSignUp) {
      setError('สมัครสำเร็จ! กรุณาตรวจอีเมลเพื่อยืนยัน')

      <div className="w-full max-w-sm p-6 bg-white rounded-xl shadow">
        <h1 className="text-2xl font-bold text-center mb-6">
          {isSignUp ? '📝 สมัครสมาชิก' : '🔑 เข้าสู่ระบบ'}
        </h1>

        <form onSubmit={handleSubmit} className="space-y-4">
          <input type="email" placeholder="อีเมล"
            value={email} onChange={e => setEmail(e.target.value)}
            required className="w-full p-3 border rounded-lg" />
          <input type="password" placeholder="รหัสผ่าน (อย่างน้อย 6 ตัว)"

            required minLength={6} className="w-full p-3 border rounded-lg" />

          {error && <p className="text-red-500 text-sm">{error}</p>}

          <button type="submit" disabled={loading}
            className="w-full p-3 bg-blue-600 text-white rounded-lg
                       hover:bg-blue-700 disabled:opacity-50">
            {loading ? '⏳ กำลังดำเนินการ...' : isSignUp ? 'สมัครสมาชิก' : 'เข้าสู่ระบบ'}
          </button>
        </form>

        <p className="text-center mt-4 text-sm text-gray-500">
          {isSignUp ? 'มีบัญชีแล้ว?' : 'ยังไม่มีบัญชี?'}

            className="text-blue-600 ml-1 hover:underline">

'use client'
import { useEffect, useState } from 'react'
import { supabase } from '@/lib/supabase'
import { useRouter } from 'next/navigation'

type Task = {
  id: string
  title: string
  completed: boolean
  created_at: string
}

export default function TasksPage() {
  const [tasks, setTasks] = useState<Task[]>([])
  const [newTitle, setNewTitle] = useState('')
  const [loading, setLoading] = useState(true)
  const router = useRouter()

  // ตรวจสอบ Login + โหลด Tasks
  useEffect(() => {
    async function init() {
      const { data: { user } } = await supabase.auth.getUser()
      if (!user) { router.push('/login'); return }
      await loadTasks()
    }
    init()
  }, [])

  async function loadTasks() {
    setLoading(true)
    const { data, error } = await supabase
      .from('tasks').select('*')
      .order('created_at', { ascending: false })
    if (!error && data) setTasks(data)
    setLoading(false)
  }

  async function addTask(e: React.FormEvent) {
    e.preventDefault()
    if (!newTitle.trim()) return
    const { data: { user } } = await supabase.auth.getUser()
    if (!user) return

    const { error } = await supabase
      .from('tasks')
      .insert({ title: newTitle.trim(), user_id: user.id })
    if (!error) { setNewTitle(''); await loadTasks() }
  }

  async function toggleTask(id: string, completed: boolean) {
    await supabase.from('tasks')
      .update({ completed: !completed }).eq('id', id)
    await loadTasks()
  }

  async function deleteTask(id: string) {
    if (!confirm('ต้องการลบ Task นี้?')) return
    await supabase.from('tasks').delete().eq('id', id)
    await loadTasks()
  }

  async function handleLogout() {
    await supabase.auth.signOut()
    router.push('/login')
  }

  return (
    <div className="max-w-lg mx-auto p-4">
      <div className="flex justify-between items-center mb-6">
        <h1 className="text-2xl font-bold">📋 Tasks</h1>
        <button onClick={handleLogout}
          className="text-sm text-red-500 hover:underline">
          ออกจากระบบ
        </button>
      </div>

      {/* ฟอร์มเพิ่ม Task */}
      <form onSubmit={addTask} className="flex gap-2 mb-6">
        <input type="text" placeholder="เพิ่ม Task ใหม่..."
          value={newTitle} onChange={e => setNewTitle(e.target.value)}
          className="flex-1 p-3 border rounded-lg" />
        <button type="submit"
          className="px-4 py-3 bg-blue-600 text-white rounded-lg">
          ➕
        </button>
      </form>

      {/* รายการ Tasks */}
      {loading ? (
        <p className="text-center text-gray-400">⏳ กำลังโหลด...</p>
      ) : tasks.length === 0 ? (
        <p className="text-center text-gray-400">📭 ยังไม่มี Task</p>
      ) : (
        <ul className="space-y-2">
          {tasks.map(task => (
            <li key={task.id}
              className="flex items-center gap-3 p-3 bg-white
                         border rounded-lg shadow-sm">
              <input type="checkbox" checked={task.completed}
                onChange={() => toggleTask(task.id, task.completed)}
                className="w-5 h-5" />
              <span className={task.completed
                ? 'flex-1 line-through text-gray-400'
                : 'flex-1'}>
                {task.title}
              </span>
              <button onClick={() => deleteTask(task.id)}
                className="text-red-400 hover:text-red-600">
                🗑️
              </button>
            </li>
          ))}
        </ul>
      )}
    </div>
  )
}

Production Best Practices — เขียนโค้ดให้ทีมอ่านง่ายและดูแลต่อได้

🟡 Intermediate ⏱️ อ่าน ~18 นาที

โค้ดที่ “ทำงานได้” ยังไม่พอสำหรับโปรเจกต์จริง เพราะโค้ดต้องอ่านง่าย แก้ง่าย ทดสอบง่าย และปลอดภัยเมื่อมีหลายคนทำงานร่วมกัน บทนี้สรุปแนวทางที่ช่วยให้โปรเจกต์เติบโตโดยไม่กลายเป็นโค้ดที่ไม่มีใครกล้าแตะ

1️⃣ Clean Code — ให้โค้ดบอกความหมายด้วยตัวเอง

Clean Code ไม่ได้หมายถึงโค้ดสั้นที่สุด แต่หมายถึงโค้ดที่คนอื่นเปิดมาแล้วเข้าใจเจตนาได้โดยไม่ต้องเดา

❌ อ่านยาก

ชื่อไม่สื่อความหมายและทำหลายอย่างพร้อมกัน
function x(a) {
  if (a) {
    if (a.user) {
      if (a.user.active) {

      }
    }
  }
}

function redirectActiveUser(session) {
  const user = session?.user

  if (!user || !user.active) return

  saveCurrentUser(user)

}

function saveCurrentUser(user) {
  localStorage.setItem(
    'current-user',
    JSON.stringify(user)
  )
}

4️⃣ Network — ตรวจ API จากหลักฐานจริง

เปิด Network แล้ว Reload หน้า จากนั้นเลือก Request ที่มีปัญหา ตรวจ URL, Method, Status, Request Payload, Response Body และ Timing

Statusความหมายโดยทั่วไปสิ่งที่ควรตรวจ
200สำเร็จResponse Shape ตรงกับที่ UI คาดหรือไม่
201สร้างข้อมูลสำเร็จRecord ถูกบันทึกและคืนค่าหรือไม่
400Request ไม่ถูกต้องPayload, Type และ Required Field
401ยังไม่ยืนยันตัวตนSession, Token และการ Login
403ยืนยันตัวตนแล้วแต่ไม่มีสิทธิ์RLS Policy และ Authorization
404ไม่พบ Route หรือ ResourceURL, Dynamic Param และ Deployment
409ข้อมูลชนกันUnique Constraint หรือข้อมูลซ้ำ
429เรียกถี่เกินไปRate Limit และ Retry Strategy
500Server ผิดพลาดServer Logs, Function Logs และ Stack Trace
Fetch ที่ตรวจ Status และ Response
async function createTask(payload) {
  const response = await fetch('/api/tasks', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json'
    },
    body: JSON.stringify(payload)
  })

  const result = await response.json()

  if (!response.ok) {
    console.error('[createTask]', {
      status: response.status,
      result
    })

    throw new Error(result.message ?? 'สร้าง Task ไม่สำเร็จ')
  }

  return result
}

async function loadPage(page: number) {
  const from = page * PAGE_SIZE
  const to = from + PAGE_SIZE - 1

  const { data } = await supabase
    .from('tasks')
    .select('*')
    .order('created_at', { ascending: false })

สร้าง Task
async function createTask(title) {
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) throw new Error('กรุณาเข้าสู่ระบบ')

  const { data, error } = await supabase
    .from('tasks')
    .insert({
      title: title.trim(),
      user_id: user.id
    })
    .select()
    .single()

  if (error) throw error
  return data
}

คำนวณ Summary
function summarizeTransactions(items) {
  return items.reduce((summary, item) => {
    const amount = Number(item.amount)

    if (item.type === 'income') {
      summary.income += amount
    } else {
      summary.expense += amount
    }

    summary.balance = summary.income - summary.expense
    return summary
  }, { income: 0, expense: 0, balance: 0 })
}

function createSlug(title) {
  return title
    .trim()
    .toLowerCase()
    .normalize('NFKD')
    .replace(/[^\p{Letter}\p{Number}\s-]/gu, '')
    .replace(/\s+/g, '-')
    .replace(/-+/g, '-')
}

คำถามที่พบบ่อย (FAQ)

❓ ต้องรู้เขียนโค้ดก่อนเรียนคู่มือนี้ไหม?

ไม่จำเป็น! คู่มือนี้ออกแบบสำหรับผู้เริ่มต้น อธิบายทุกขั้นตอน แต่ถ้ามีพื้นฐาน HTML/CSS/JavaScript มาก่อนจะเข้าใจเร็วขึ้น

❓ ใช้ AI เขียนโค้ดทั้งหมดได้ไหม?

ได้ แต่ต้องอ่านโค้ดก่อนใช้เสมอ ตรวจ Security (RLS, Env Vars) ให้ดี และทดสอบก่อน Deploy อย่าเชื่อ AI 100% โดยไม่ตรวจสอบ

❓ ทุกอย่างฟรีจริงหรือ?

ใช่! GitHub (Free), Vercel (Hobby plan), Supabase (Free tier — 2 โปรเจกต์, 500MB) เพียงพอสำหรับเรียนรู้และโปรเจกต์เล็ก-กลาง

❓ เลือก Next.js ทำไม? ใช้ React ธรรมดาได้ไหม?

ใช้ได้ แต่ Next.js มีข้อดีเพิ่มเติม: routing อัตโนมัติ, Server-Side Rendering, API Routes ในตัว และ Vercel สร้าง Next.js เองจึงรองรับดีที่สุด

❓ Push .env ขึ้น GitHub ไปแล้วทำอย่างไร?

1) เปลี่ยน Key/Secret ทันที (สำคัญที่สุด!) 2) เอาไฟล์ออก: git rm --cached .env 3) เพิ่ม .gitignore 4) อัปเดต Key ใหม่ทุกที่ ดูรายละเอียดในบท 🐙 GitHub

❓ เว็บ Deploy แล้วแต่หน้าขาว?

สาเหตุหลัก: Framework ถูกตรวจจับผิด, Root Directory ผิด, หรือ Build error ที่ไม่ได้ดูใน Logs — ไปที่ Vercel Dashboard → Deployments → อ่าน Build Logs

❓ ผู้ใช้คนหนึ่งเห็นข้อมูลของอีกคน?

🚨 RLS อาจยังไม่ได้เปิด หรือ Policy ตั้งผิด! ตรวจว่า: 1) RLS เปิดทุกตาราง 2) Policy ใช้ auth.uid() = user_id 3) ไม่มี USING (true) ใน SELECT

❓ จะเรียนอะไรต่อ?

หลังจบคู่มือนี้แนะนำ: TypeScript เชิงลึก → React Hooks → Next.js App Router → Tailwind CSS → PostgreSQL → Testing
